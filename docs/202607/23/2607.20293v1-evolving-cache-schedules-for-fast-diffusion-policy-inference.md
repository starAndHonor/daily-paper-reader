## 摘要
扩散策略通过迭代去噪动作块实现强大的视觉运动控制，但重复去噪使得实时部署的计算需求很高。基于缓存的方法通过重用中间激活来降低推理成本，但现有的无需训练调度通常跨模块均匀分配计算，忽略了模块间的异构冗余，导致次优的性能-效率权衡。为弥合这一差距，我们提出进化缓存调度（EVO），一种无需训练的加速框架，通过进化搜索全局调度缓存刷新。EVO将每个候选表示为一个在块-时间步网格上的完整调度。因此，在迭代去噪过程中，可以通过缓存重用跳过冗余的Transformer计算，同时保持闭环滚动性能。为了使搜索实用，EVO引入了冗余感知初始化，用有前景的调度播种种群，以及目标条件早停，一旦达到期望的性能目标即验证并终止。离线优化的调度可直接插入预训练的扩散策略中，无需重新训练。广泛的操作基准测试表明，EVO在显著减少计算的同时保持了接近完整的性能，实现了高达8.05倍的动作生成加速，并将FLOPs从15.77G降低至1.96G。源代码可在https://github.com/pillom/EVO获取。

## Abstract
Diffusion policies achieve strong visuomotor control by iteratively denoising action chunks, but repeated denoising makes real-time deployment computationally demanding. Cache-based methods reduce inference cost by reusing intermediate activations, but existing training-free schedules typically allocate computation uniformly across blocks, ignoring heterogeneous redundancy across blocks and leading to a suboptimal performance-efficiency trade-off. To bridge this gap, we introduce Evolving Cache Schedules (EVO), a training-free acceleration framework that globally schedules cache refreshes via evolutionary search. EVO represents each candidate as a complete schedule over the block-timestep lattice. Thus, redundant transformer computations during iterative denoising can be skipped through cache reuse while preserving closed-loop rollout performance. To make the search practical, EVO introduces redundancy-aware initialization, which seeds the population with promising schedules, and target-conditioned early stopping, which verifies and terminates once a desired performance target is reached. The offline-optimized schedule can be directly plugged into pretrained diffusion policies without retraining. Extensive manipulation benchmarks show that EVO preserves near-full performance while substantially reducing computation, achieving up to 8.05x action-generation speedup and reducing FLOPs from 15.77G to as low as 1.96G. Source code is available at https://github.com/pillom/EVO.

# 论文总结：Evolving Cache Schedules for Fast Diffusion Policy Inference

## 1. 核心问题与整体含义（研究动机和背景）

扩散策略（Diffusion Policy）利用迭代去噪生成动作块，在视觉运动控制中表现出强大能力，但重复的Transformer去噪计算导致高延迟，难以满足实时机器人控制的低延迟需求。现有基于缓存的加速方法（如EfficientVLA、BAC）虽然能重用中间激活减少计算，但它们通常对每个Transformer块分配均匀的刷新预算，忽略了不同块在去噪过程中冗余程度的差异。论文通过实验发现：不同Transformer块（自注意力、交叉注意力、FFN）在时间维度上呈现异构冗余模式，且特征相似性与任务性能下降之间仅存在弱相关性。因此，**核心问题**是如何在固定计算预算下，全局优化块-时间步网格上的缓存刷新位置，以在保持闭环控制性能的同时最大化加速效果。整体含义：提出一种无需重训练的全局缓存调度搜索框架，可直接嵌入预训练模型，显著降低推理成本。

## 2. 方法论

### 核心思想
将缓存调度建模为一个固定预算的子集选择问题：在块集合 \(B\) 和时间步集合 \(T\) 组成的格点 \(B\times T\) 上，选择 \(K\) 个位置执行原始残差计算（刷新缓存），其余位置重用最接近的缓存激活。目标是最大化闭环 rollout 成功率。

### 关键技术细节
- **全局缓存调度**：每个去噪步骤中，每个缓存单元（自注意力、交叉注意力、FFN残差分支）的残差计算可被选择刷新或重用。调度 \(S\subseteq B\times T, |S|=K\) 决定哪些位置执行计算。
- **进化搜索**：将每个候选调度表示为一个包含 \(K\) 个唯一刷新位置的个体。使用锦标赛选择、集合级交叉、变异、修复和精英保留策略进化。适应度函数为 rollout 成功率（每次评估需要多轮环境交互）。
- **冗余感知初始化**：从完整策略收集少量 rollouts，计算每个块-时间步位置的跨步特征不相似度（cosine dissimilarity），构成概率分布，对低冗余位置赋予更高采样权重，用于初始化部分种群。但仍保留随机个体以维持多样性。
- **目标条件早停**：快速评估（20 episodes）筛选候选，若达到基线性能则进行正式评估（50 episodes），通过后直接终止搜索，避免冗余计算。

### 算法流程（文字说明）
1. **初始化**：生成初始种群（部分由冗余感知引导，部分随机）。
2. **适应度评估**：对每个候选调度，在仿真环境中运行固定数量 episodes，计算成功率。
3. **进化迭代**：选择 → 交叉 → 变异 → 修复 → 精英保留 → 生成下一代。
4. **早停检查**：每代后检查是否有候选通过正式评估（快速评估 + 独立正式评估），若满足则终止并输出当前最佳调度。
5. **部署**：将最终调度作为离线策略，在测试时直接加载，无需在线搜索或重训练。

1. EVO在保持近满性能（平均成功率0.78~0.99 vs. 完整模型0.80~0.99）的同时，实现了**高达8.05倍动作生成加速**，FLOPs从15.77G降至1.96G。

5. 目标条件早停可有效减少离线搜索成本，避免过早终止。

## 7. 优点

- **无需重训练**：直接作用于预训练模型，节省大量训练成本。
- **全局搜索**：首次将缓存调度建模为格点上的全局面包屑分配问题，而非局部规则，更符合异构冗余现实。
- **搜索效率**：冗余感知初始化 + 目标条件早停 + 去重缓存，使基于rollout的演化搜索在可接受时间内可行（<1 GPU-day）。
- **系统消融**：严格验证了每块分配 vs. 全局分配、相似度初始化 vs. 随机、不同相似度度量等设计选择。
- **代码开源**：提供源代码，可复现。

## 8. 不足与局限

- **仅仿真环境验证**：所有实验在模拟器中完成，未在真实机器人上验证，实用中可能存在sim-to-real gap（观测噪声、延迟等）。
- **未与重训练方法对比**：如扩散蒸馏（1-step diffusion policy）、一致性模型等，这些方法通常性能更好但需训练。EVO作为免训练方法，优势在于即插即用，但性能边界未与重训练方法对比。
- **搜索成本相对高**：虽然单次搜索成本在1 GPU-day以内，但对于每个新任务仍需离线搜索。任务分布变化大时可能需要重新搜索（文中只验证了同分布重用）。
- **评价指标局限**：主要使用成功率，缺乏细粒度指标（如动作误差、轨迹平滑度），加速比基于理论FLOPs，实际推理时间因硬件和实现而异。
- **未考虑缓存内存开销**：虽然缓存重用减少计算，但需要额外存储中间激活，文中未分析内存占用。
- **可扩展性**：搜索空间随块数和时间步增长，对于更大模型（如DiT）或更多步数，搜索成本可能非线性增长，文中仅使用8层、100步。
- **评估方差**：成功率在某些任务上标准差较高（如Transport PH 0.39±0.08），性能波动可能受rollout随机性影响，需更多种子验证。
